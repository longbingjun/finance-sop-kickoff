# 参考细节

## §1 环节切分信号详解

| 信号类型 | 具体表现 | 说明 |
|----------|----------|------|
| 动词切换 | 叙述中动作动词聚类切换：筛选/剔除 → 匹配/关联 → 汇总 → 核对 → 导出 | 动词类别整体切换处，可能是环节边界 |
| 文件切换 | 叙述或文件夹中出现"从A表到B表"的转换节点 | 产出了新的中间文件，是较强信号 |
| Sheet/Tab切换 | Excel 内部 sheet 命名本身隐含分段（"原始数据""核对""汇总"） | **权重最高**——财务人员日常已用 tab 隐性分段，比口头叙述更可靠、更容易被观察到 |
| 统计粒度切换 | "一行代表的东西"发生变化（订单行 → 品号行 → 型号行） | 与 03 口径卡的统计对象概念直接相关 |
| 交接信号 | 叙述中出现"这份要给XX""这里要等XX的数据" | 跨人、跨部门、跨系统的移交点，通常对应 02 的同步对象 |
| 已有目录/编号信号 | 项目文件夹里已有命名规律（哪怕不规范，如"1-初步整理" "2-核对表"） | 直接沿用，不需要重新归纳 |

## §2 字段优先级分级规则

### 字段类型判定

| 类型 | 判定方法 |
|------|----------|
| 数值/度量型 | dtype 为数值，且非纯序号/ID |
| 分类/代码型 | 唯一值数量 ≤ 上限（默认20），或字段名匹配编码模式 |
| 自由文本型 | dtype 为字符串，且"唯一值数/非空行数"比例较高、平均字符串长度较长 |
| 疑似主键/ID | 唯一值数 ≈ 非空行数，且不符合自由文本模式 |

### 优先级判定

| 优先级 | 触发条件 |
|--------|----------|
| 高 | 数值/分类型字段空值率超阈值（默认5%）；数值型存在 IQR 之外的极端值；字段名是缩写/代码难以望文生义；字段是衍生/计算字段；环节拆解阶段已被提及为筛选条件 |
| 中 | 分类型字段取值未确认；疑似主键字段（需确认"一行代表什么"） |
| 低 | 自由文本字段；命名清晰且未被提及为关键字段的字段 |

**重要修正**：自由文本型字段的高空值率不计入"高优先级"判定——备注类字段本来就该大量为空，这不是异常，是正常。判断字段是否属于自由文本型要先于判断空值率是否异常。

## §3 异常阈值默认参数（脚本可调）

- 空值率高优先级阈值：`5%`（`scan_fields.py --null-threshold`）
- 数值型极端值判定：IQR 方法，超出 `[Q1 - 1.5×IQR, Q3 + 1.5×IQR]` 视为潜在极端值
- 分类型低基数上限：`20` 个唯一值以内视为分类型，超过视为自由文本或 ID 候选（`scan_fields.py --max-categories`）

这些是脚本用来生成"待确认清单"优先级的默认参数，不是业务口径本身——业务口径里的"正常范围/异常阈值"必须来自访谈中业务人员的确认，脚本参数只决定"先问哪个字段"，不能替代业务判断写入 03 文档。

## §4 04文档的异常归因分类

04 台账"异常归因"列不要只写"人的问题"，按下面六类具体归因，帮助后续同类问题快速定位该找谁改：

| 归因类别 | 含义 | 举例 |
|----------|------|------|
| 人 | 某次具体的个人失误（一次性，非制度性） | 某次手动录入打错了一个数字 |
| 组织 | 跨部门协作机制、职责边界不清导致的问题 | 两个部门都以为对方会核对某个字段 |
| 流程 | 环节设计本身有缺陷（不是人的问题，换个人还是会出错） | 某一步骤没有校验环节，错误无法被发现 |
| 系统 | IT 系统/工具本身的限制或 bug | 导出的数据格式在某个系统版本后变了 |
| 能力 | 负责人对业务/工具不够熟练导致的错误 | 不熟悉某个函数导致公式用错 |
| 态度 | 并非能力问题，是重视程度/认真程度问题 | 明知道该核对但没核对 |

**"态度"这一类要谨慎使用**——必须有客观、可复述的事实支撑（比如"同一个问题提醒过still没改"），不能只凭印象归类为态度问题，避免台账变成对人的主观评价记录，与04文档"不对人贴标签"的原则保持一致。
